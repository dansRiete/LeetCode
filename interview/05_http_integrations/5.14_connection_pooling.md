## Complete Answer

### Why connection pools exist:
Creating a connection is expensive: TCP handshake → TLS handshake → DB auth → session setup.
Pool creates connections once and reuses them.

### HikariCP key config:
```java
config.setConnectionTimeout(30000);    // wait for pool slot (ms)
config.setMaximumPoolSize(10);
config.setMinimumIdle(2);
config.setIdleTimeout(600000);
config.setMaxLifetime(1800000);
config.setLeakDetectionThreshold(2000); // warn if held > 2s
```

### Connection leak — safe pattern:
```java
try (Connection conn = pool.getConnection();
     PreparedStatement ps = conn.prepareStatement(sql)) {
    ps.execute();
} // conn.close() returns connection to pool — not actually closed
```

### HTTP Keep-Alive:
- HTTP/1.1 default: `Connection: keep-alive`
- Reuses TCP socket across requests — avoids repeated TCP+TLS overhead
- Common mistake: `new HttpClient()` per request defeats the pool

### HTTP/1.1 vs HTTP/2:
| Feature | HTTP/1.1 | HTTP/2 |
|---------|----------|--------|
| Connections needed | 6+ per host | 1 |
| Multiplexing | No (head-of-line blocking) | Yes |
| Header compression | No | Yes (HPACK) |
| Binary protocol | No | Yes |

### Connection vs Read timeout:
- **Connection timeout** — time to establish TCP connection. Fires if server unreachable.
- **Read timeout** — time waiting for data after connection established. Fires if server is slow.
- Slow DB scenario: connection succeeds → read timeout fires → `SocketTimeoutException: Read timed out`
