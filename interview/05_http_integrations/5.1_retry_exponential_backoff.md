## Complete Answer

### Plain Java implementation:
```java
public <T> T executeWithRetry(Supplier<T> task, int maxRetries)
        throws InterruptedException {
    int attempt = 0;
    while (true) {
        try {
            return task.get();
        } catch (TransientException e) {
            attempt++;
            if (attempt >= maxRetries) throw e;

            long waitMs = calculateBackoff(attempt);
            Thread.sleep(waitMs);
        } catch (PermanentException e) {
            throw e; // never retry permanent failures
        }
    }
}

private long calculateBackoff(int attempt) {
    long base = (long) Math.pow(2, attempt) * 100; // 200ms, 400ms, 800ms, 1600ms...
    long jitter = (long) (Math.random() * 100);    // add randomness
    return Math.min(base + jitter, 30_000);        // cap at 30 seconds
}
```

### Backoff progression:

- Attempt 1: base 200ms, with jitter ~200-300ms, capped at 30s
- Attempt 2: base 400ms, with jitter ~400-500ms, capped at 30s
- Attempt 3: base 800ms, with jitter ~800-900ms, capped at 30s
- Attempt 4: base 1600ms, with jitter ~1600-1700ms, capped at 30s
- Attempt 10: base 102400ms, capped at **30000ms**

### Why jitter matters — thundering herd:
Without jitter: 100 instances all retry at exactly 200ms, 400ms, 800ms →
all hammer the partner simultaneously.
With jitter: retries are spread out → partner recovers more easily.

### What to retry vs not:
```java
private boolean isRetryable(int statusCode) {
    return statusCode == 429   // rate limited
        || statusCode == 503   // service unavailable
        || statusCode == 504   // gateway timeout
        || statusCode >= 500;  // server errors
}
// never retry 4xx (except 429) — permanent failures
```
