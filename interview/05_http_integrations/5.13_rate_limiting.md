# 32. How do you handle a partner API that is rate-limited (429 responses)?

## Core Idea
Rate limiting means the partner is saying "you're calling us too fast, slow down."
The goal is to **stay within the partner's limits proactively** rather than hitting
429 reactively.

---

## Level 1 — Reactive: Handle 429 When It Happens
```java
if (statusCode == 429) {
    // respect Retry-After header if partner sends one
    String retryAfter = response.getHeader("Retry-After");
    long waitMs = retryAfter != null
        ? Long.parseLong(retryAfter) * 1000
        : calculateExponentialBackoff(attempt);

    Thread.sleep(waitMs);
    return retry(request, attempt + 1);
}
```

---

## Level 2 — Proactive: Rate Limiter on Your Side

Don't wait for 429 — control your own call rate using a token bucket or fixed window:

### Simple Fixed Window Rate Limiter
```java
public class RateLimiter {
    private final int maxCallsPerSecond;
    private int callsThisSecond = 0;
    private long windowStart = System.currentTimeMillis();

    public synchronized void acquire() throws InterruptedException {
        long now = System.currentTimeMillis();
        if (now - windowStart >= 1000) {
            callsThisSecond = 0;
            windowStart = now;
        }
        if (callsThisSecond >= maxCallsPerSecond) {
            long waitMs = 1000 - (now - windowStart);
            Thread.sleep(waitMs);
            callsThisSecond = 0;
            windowStart = System.currentTimeMillis();
        }
        callsThisSecond++;
    }
}

// usage
rateLimiter.acquire();
partnerApi.call();
```

### Using ScheduledExecutorService for Batching
```java
// instead of calling as fast as possible, schedule N calls per second
ScheduledExecutorService scheduler = Executors.newScheduledThreadPool(1);
AtomicInteger index = new AtomicInteger(0);

scheduler.scheduleAtFixedRate(() -> {
    int i = index.getAndIncrement();
    if (i < records.size()) {
        processRecord(records.get(i));
    }
}, 0, 100, TimeUnit.MILLISECONDS); // one record every 100ms = max 10/second
```

## Level 3 — Queue Based for High Volume
```java
// put work on a bounded queue
BlockingQueue<Request> workQueue = new ArrayBlockingQueue<>(1000);

// single worker thread that respects rate limit
executor.submit(() -> {
    while (true) {
        Request request = workQueue.take();
        rateLimiter.acquire();
        partnerApi.call(request);
    }
});
```

---

## Key Things to Know Per Partner
- What is the rate limit? (calls/second, calls/minute, calls/day)
- Is it per endpoint or global?
- Does the partner send `Retry-After` or `X-RateLimit-Remaining` headers?
- Are there different limits for different API tiers?

---

## Senior Points to Mention
- Always read partner documentation for rate limits before writing the connector
- Implement proactive rate limiting — reactive (waiting for 429) wastes time and
  causes jitter
- Monitor your call rate as a metric — alert before you approach the limit
- For burst scenarios (e.g. batch processing) — use a queue to smooth out the
  call rate
- `Retry-After` header — always respect it, ignoring it can get your API key banned
- Different environments (prod vs staging) may have different rate limits
