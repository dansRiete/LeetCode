# 28. Explain the Circuit Breaker pattern — states, transitions, and when to use it

## Core Idea
Prevents your connector from repeatedly calling a partner API that is clearly failing,
giving it time to recover while protecting your system from cascading failures.

Named after an electrical circuit breaker — trips when too much current flows,
cutting the circuit.

---

## Three States

### CLOSED (normal operation)
- Requests flow through normally
- Failures are counted
- If failure rate exceeds threshold (e.g. 50% of last 10 calls) → trips to OPEN

### OPEN (partner is down)
- Requests are **immediately rejected** without calling the partner
- Returns a fallback response or throws an exception right away
- After a timeout period (e.g. 30s) → moves to HALF_OPEN

### HALF_OPEN (recovery probe)
- Allows **one test request** through
- If it succeeds → back to CLOSED (partner recovered)
- If it fails → back to OPEN (partner still down)

---

## State Diagram
```
CLOSED → (failure threshold reached) → OPEN
OPEN → (timeout expires) → HALF_OPEN
HALF_OPEN → (test request succeeds) → CLOSED
HALF_OPEN → (test request fails) → OPEN
```

---

## Plain Java Implementation
```java
public class CircuitBreaker {
    enum State { CLOSED, OPEN, HALF_OPEN }

    private State state = State.CLOSED;
    private int failureCount = 0;
    private long openedAt;

    private static final int FAILURE_THRESHOLD = 5;
    private static final long TIMEOUT_MS = 30_000;

    public <T> T execute(Supplier<T> call, Supplier<T> fallback) {
        if (state == State.OPEN) {
            if (System.currentTimeMillis() - openedAt > TIMEOUT_MS) {
                state = State.HALF_OPEN;
            } else {
                return fallback.get(); // fail fast
            }
        }

        try {
            T result = call.get();
            onSuccess();
            return result;
        } catch (Exception e) {
            onFailure();
            return fallback.get();
        }
    }

    private void onSuccess() {
        failureCount = 0;
        state = State.CLOSED;
    }

    private void onFailure() {
        failureCount++;
        if (failureCount >= FAILURE_THRESHOLD || state == State.HALF_OPEN) {
            state = State.OPEN;
            openedAt = System.currentTimeMillis();
        }
    }
}
```

---

## When to Use
- Partner API is intermittently down
- Slow partner responses are cascading into your system
- You want to give a degraded but functional response instead of hanging

## Senior Points to Mention
- Without circuit breaker: threads pile up waiting for a dead partner, exhausting
  your thread pool
- Fallback options: cached response, default value, error message to user
- Monitor state transitions as metrics — frequent OPEN state = partner SLA problem
- Combine with **retry + backoff** for transient failures, circuit breaker for
  sustained failures
