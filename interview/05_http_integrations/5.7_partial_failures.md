# 19. How do you handle partial failures — partner returns 200 but with error fields in the body?

## Complete Answer

### Modeling the response:
```java
@JsonIgnoreProperties(ignoreUnknown = true)
public class PartnerResponse {
    private List<BookingResult> results;

    public record BookingResult(
        String id,
        String status,       // "SUCCESS" or "FAILED"
        String errorCode,    // null if successful
        String errorMessage  // null if successful
    ) {}
}
```

### Processing partial failures:
```java
Map<Boolean, List<BookingResult>> partitioned = response.getResults()
    .stream()
    .collect(Collectors.partitioningBy(r -> "SUCCESS".equals(r.status())));

List<BookingResult> succeeded = partitioned.get(true);
List<BookingResult> failed = partitioned.get(false);

process(succeeded);

// separate retryable from permanent failures
failed.forEach(result -> {
    if (isRetryable(result.errorCode())) {
        retryQueue.add(result);
    } else {
        deadLetterQueue.add(result, result.errorMessage(), 0);
    }
});
```

### Retryable vs permanent error codes:
```java
private boolean isRetryable(String errorCode) {
    return switch (errorCode) {
        case "DOWNSTREAM_TIMEOUT", "PARTNER_UNAVAILABLE" -> true;  // transient
        case "INVALID_BOOKING_ID", "DUPLICATE_REQUEST"   -> false; // permanent
        default -> false; // unknown = treat as permanent, investigate
    };
}
```

### Alerting on high failure rate:
```java
double failureRate = (double) failed.size() / response.getResults().size();
if (failureRate > 0.1) { // more than 10% failing
    alert("High failure rate from partner: " + failureRate);
}
```

### Senior points:
- Never silently discard failed records — always DLQ or alert
- Document which error codes each partner returns — they're inconsistent
- Idempotency — before retrying confirm reprocessing won't cause duplicates on
  partner side
- Log failed records with full context for debugging
