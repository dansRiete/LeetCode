# 31. How do you distinguish between a transient failure (retry) and a permanent failure (fail fast)?

## Core Rule
- **Transient** — temporary condition, retrying may succeed.
  The data is fine, the system is not.
- **Permanent** — retrying will never help.
  Either the data is wrong or you lack permission.

---

## By HTTP Status Code

### Always Retry (Transient)

- 408 Request Timeout — server was busy, try again
- 429 Too Many Requests — rate limited, wait and retry
- 500 Internal Server Error — partner had a bug, may recover
- 502 Bad Gateway — upstream issue, temporary
- 503 Service Unavailable — partner is down, temporary
- 504 Gateway Timeout — timeout in partner's infrastructure

### Never Retry (Permanent)

- 400 Bad Request — your request is malformed, fix the data
- 403 Forbidden — no permission, retrying won't grant access
- 404 Not Found — resource doesn't exist
- 422 Unprocessable Entity — validation failed, fix the data

### Special Case — 401 Unauthorized
```java
// 401 can be either:
// 1. Expired OAuth token → refresh token and retry ONCE
// 2. Wrong credentials → permanent failure, don't retry

if (statusCode == 401 && !hasRetriedTokenRefresh) {
    refreshToken();
    return retry(request, true); // retry once after refresh
}
// if still 401 after refresh → permanent failure
```

## By Network Exception
```java
catch (SocketTimeoutException e)  → transient, retry
catch (ConnectException e)        → transient, retry
catch (UnknownHostException e)    → transient, retry with backoff
catch (SSLException e)            → permanent, cert issue needs fixing
```

---

## By Business Error in 200 Response
```java
// partner returns 200 but error in body
switch (response.getErrorCode()) {
    case "DOWNSTREAM_TIMEOUT"   -> retry;       // transient
    case "INVALID_BOOKING_ID"   -> permanent;   // data issue
    case "DUPLICATE_REQUEST"    -> permanent;   // idempotency issue
    case "PARTNER_UNAVAILABLE"  -> retry;       // transient
}
```

---

## Implementation
```java
private boolean isRetryable(Exception e, int statusCode) {
    if (e instanceof SocketTimeoutException) return true;
    if (e instanceof ConnectException) return true;
    return statusCode == 429
        || statusCode == 503
        || statusCode == 504
        || statusCode == 500;
}

private boolean isPermanent(int statusCode) {
    return statusCode == 400
        || statusCode == 403
        || statusCode == 404
        || statusCode == 422;
}
```

---

## Senior Points to Mention
- Log permanent failures with full context — they need investigation
- Send permanent failures to DLQ immediately — no point retrying
- For 429 — respect the `Retry-After` header if partner sends one
- Never retry without backoff — hammering a struggling partner makes it worse
- Document which error codes each partner returns — partners are inconsistent
