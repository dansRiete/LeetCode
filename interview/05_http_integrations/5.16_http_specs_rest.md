## Complete Answer

### HTTP Methods:
| Method | Safe | Idempotent |
|--------|------|------------|
| GET | ✓ | ✓ |
| POST | ✗ | ✗ |
| PUT | ✗ | ✓ |
| DELETE | ✗ | ✓ |
| PATCH | ✗ | ✗ |

- **Safe** = no state change (read-only)
- **Idempotent** = same result regardless of how many times called

### PUT vs PATCH:
- PUT = full replacement (or create if client knows ID). Missing fields → nulled.
- PATCH = partial update, only specified fields change

### Status code categories:
- 1xx — Informational (100 Continue)
- 2xx — Success (200 OK, 201 Created, 204 No Content)
- 3xx — Redirection (301 Moved, 304 Not Modified)
- 4xx — Client error (400, 401, 403, 404, 409 Conflict, 429 Rate Limited)
- 5xx — Server error (500, 502 Bad Gateway, 503 Unavailable, 504 Gateway Timeout)

### 401 vs 403:
- 401 — not authenticated ("I don't know who you are") → retry with credentials
- 403 — authenticated but not authorized ("you can't do this") → credentials won't help

### REST constraints (Roy Fielding):
| Constraint | Meaning |
|-----------|---------|
| Client-Server | UI and backend separate |
| **Stateless** | No server session — every request self-contained |
| Cacheable | Responses declare cacheability |
| Uniform Interface | Consistent URLs, HTTP methods, resource-based |
| Layered System | Client unaware of proxies/gateways |
| Code on Demand | Server can send executable code (optional) |
