# 29. What is the Bulkhead pattern and why is it important for integrations?

## Core Idea
Isolate resources (threads, connections) per partner so one slow/failing partner
cannot consume all resources and bring down integrations with other partners.

Named after ship bulkheads — watertight compartments that prevent one breach from
sinking the whole ship.

---

## The Problem Without Bulkhead
```
Thread Pool: 20 threads total

Partner A goes slow → takes 18 threads waiting
Partner B, C requests → only 2 threads left → starved → appear down
```
One bad partner takes down all your integrations.

---

## The Solution — Dedicated Thread Pools Per Partner
```java
// each partner gets its own bounded thread pool
ExecutorService partnerAPool = Executors.newFixedThreadPool(5);
ExecutorService partnerBPool = Executors.newFixedThreadPool(5);
ExecutorService partnerCPool = Executors.newFixedThreadPool(5);

// Partner A goes slow — only its 5 threads are affected
// Partner B and C still have their own 5 threads each
CompletableFuture.supplyAsync(() -> partnerA.call(), partnerAPool);
CompletableFuture.supplyAsync(() -> partnerB.call(), partnerBPool);
CompletableFuture.supplyAsync(() -> partnerC.call(), partnerCPool);
```

---

## Also Applies to Connection Pools
```java
// separate DB/HTTP connection pools per partner
HikariConfig partnerAConfig = new HikariConfig();
partnerAConfig.setMaximumPoolSize(10); // Partner A gets max 10 connections

HikariConfig partnerBConfig = new HikariConfig();
partnerBConfig.setMaximumPoolSize(10); // Partner B gets max 10 connections
```

---

## Why Important for Integration Services
- A typical integration platform connects to many partners simultaneously
  (hotels, flights, cars, activities)
- One slow partner (e.g. hotel API during peak season) must not degrade flight
  or car searches
- Bulkhead ensures **fault isolation** — failures stay contained

---

## Senior Points to Mention
- Size each pool based on partner SLA and expected load — not one size fits all
- Monitor queue depth per pool — if Partner A pool queue is always full, increase
  its size or investigate partner
- Combine with **Circuit Breaker** — bulkhead isolates resources, circuit breaker
  stops calling a dead partner
- **Semaphore-based bulkhead** — alternative to thread pools, limits concurrent
  calls without dedicated threads:

```java
Semaphore partnerASemaphore = new Semaphore(5); // max 5 concurrent calls to Partner A

try {
    partnerASemaphore.acquire();
    return partnerA.call();
} finally {
    partnerASemaphore.release();
}
```
