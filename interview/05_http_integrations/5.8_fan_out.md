# 20. How would you design a connector that needs to fan out to 5 partner APIs and aggregate results?

## Your Answer
- Run each in a separate thread, then join() to wait for all ✓
- Set timeouts on each call ✓
- Retry on timeout with exponential backoff ✓
- Partial vs full failure depends on business logic ✓

## What Was Missing / Wrong
- Conceptually correct but missing modern Java approach
- Missing: `CompletableFuture` — cleaner than raw threads
- Missing: dedicated `ExecutorService` — don't use the common ForkJoinPool for I/O

## Complete Answer

### Fan out = send request to multiple partners simultaneously, aggregate results:
```
                    → Partner API 1 (flights)
                    → Partner API 2 (hotels)
Request → Connector → Partner API 3 (cars)     → Aggregate → Response
                    → Partner API 4 (activities)
                    → Partner API 5 (insurance)
```

### CompletableFuture implementation:
```java
private final ExecutorService partnerPool = Executors.newFixedThreadPool(10);

public AggregatedResult fanOut(List<PartnerClient> partners) {
    List<CompletableFuture<PartnerResult>> futures = partners.stream()
        .map(partner -> CompletableFuture
            .supplyAsync(() -> partner.call(), partnerPool)  // dedicated pool
            .orTimeout(5, TimeUnit.SECONDS)                  // per-call timeout
            .exceptionally(ex -> PartnerResult.failed(ex.getMessage())))
        .toList();

    // wait for ALL to complete
    List<PartnerResult> results = futures.stream()
        .map(CompletableFuture::join)
        .toList();

    // business decision — partial or full failure
    List<PartnerResult> failed = results.stream()
        .filter(PartnerResult::isFailed)
        .toList();

    if (!failed.isEmpty() && requiresAllResults) {
        throw new PartialFailureException(failed);
    }

    return aggregate(results.stream().filter(PartnerResult::isSuccess).toList());
}
```

### Why CompletableFuture over raw threads:
- No manual thread management
- `.orTimeout()` — built-in per-call timeout
- `.exceptionally()` — handle individual failures without failing all futures
- Composable — can chain transformations

### Why dedicated ExecutorService:
- `parallelStream()` uses shared ForkJoinPool — blocking I/O calls starve other tasks
- Dedicated pool: size it based on number of partners × expected concurrency

### Senior points:
- Size thread pool = number of concurrent fan-outs × partners per fan-out
- Cache partner clients — don't create new HTTP connections per request
- Add per-partner circuit breakers — one slow partner shouldn't affect others
  (Bulkhead pattern)
- Return partial results with metadata — which partners succeeded, which failed, why
