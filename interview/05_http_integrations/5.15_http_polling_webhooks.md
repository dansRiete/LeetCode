## Complete Answer

### Two polling use cases:
1. **Async job polling** — poll `GET /job/{id}/status` until COMPLETED/FAILED
2. **Real-time update polling** — poll `GET /events` every N seconds for new data

### Polling drawbacks:
- Most calls return nothing — wasted resources
- Unnecessary load on partner → rate limiting (429)
- Not truly real-time — latency baked in by interval
- Wastes threads on your side

### Alternatives:
| Mechanism | How | When |
|-----------|-----|------|
| Long polling | Server holds request open until event occurs | Simple push without WebSocket |
| WebSocket | Persistent bidirectional connection | High-frequency real-time |
| SSE | Server pushes over persistent HTTP (one-way) | Server→client stream |
| Webhooks | Partner POSTs to your endpoint on event | Integration gold standard |

### Webhook security — HMAC signature:
```java
// Partner sends: X-Partner-Signature: sha256=abc123...
String expected = hmacSHA256(secret, requestBody);
String received = request.getHeader("X-Partner-Signature");

// Use constant-time comparison to prevent timing attacks
if (!MessageDigest.isEqual(expected.getBytes(), received.getBytes())) {
    throw new SecurityException("Invalid webhook signature");
}
```

### Additional webhook security:
- Validate timestamp — reject requests older than 5 minutes (replay attack prevention)
- HTTPS only
- Return 200 immediately, process async — prevent partner retries from slow processing
