# 15. How do you deserialize a JSON response when the schema may change between partner versions?

## Your Answer
- Try-catch with new schema as fallback ✗

## What Was Missing / Wrong
- Try-catch for control flow is an anti-pattern — exceptions are expensive
- The right approach is **prevention**, not recovery
- Missing: `FAIL_ON_UNKNOWN_PROPERTIES=false` — most important answer
- Missing: versioned DTOs
- Missing: contract tests

## Complete Answer

### Strategy 1 — Lenient deserialization (most important):
```java
// globally on ObjectMapper — apply once, works for all DTOs
ObjectMapper mapper = new ObjectMapper();
mapper.configure(DeserializationFeature.FAIL_ON_UNKNOWN_PROPERTIES, false);

// or per class with annotation
@JsonIgnoreProperties(ignoreUnknown = true)
public class FlightResponse {
    private String destination;
    // partner adds "newField" → silently ignored, no crash
}
```

### Strategy 2 — Nullable fields for fields that may disappear:
```java
public class FlightResponse {
    private String destination;   // always present
    private String promoCode;     // may not exist in older versions — null if absent
}
```

### Strategy 3 — Versioned DTOs for major changes:
```java
class PartnerResponseV1 { String destination; int price; }
class PartnerResponseV2 { String destination; int price; String currency; }

// route based on API version header or endpoint
String version = response.getHeader("X-Api-Version");
if ("v2".equals(version)) {
    return mapper.readValue(body, PartnerResponseV2.class);
} else {
    return mapper.readValue(body, PartnerResponseV1.class);
}
```

### Strategy 4 — Dynamic parsing for truly unknown schemas:
```java
JsonNode root = mapper.readTree(response);
String dest = root.path("destination").asText("");  // never throws, returns "" if missing
int price = root.path("price").asInt(0);            // never throws, returns 0 if missing
```

### Senior points:
- **Prevention > recovery** — configure deserializer to be lenient upfront
- Agree on **versioning strategy** with partner before integration starts
- **Contract tests** (e.g. Pact) — detect schema changes in CI before they hit production
- Never use try-catch for expected schema variations — it's slow and masks real errors
