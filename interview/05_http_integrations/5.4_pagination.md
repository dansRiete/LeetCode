## Complete Answer

### Two pagination styles:

**Offset/Page based:**
```
GET /bookings?page=1&size=100
GET /bookings?page=2&size=100
```
- Simple, most common
- Problem: records added/deleted between pages → duplicates or missed records

**Cursor/Token based:**
```
GET /bookings?limit=100
Response: { "data": [...], "nextCursor": "abc123" }

GET /bookings?limit=100&cursor=abc123
Response: { "data": [...], "nextCursor": "def456" }

GET /bookings?limit=100&cursor=def456
Response: { "data": [...], "nextCursor": null } ← last page
```
- More reliable — cursor points to exact position
- No duplicates or missed records
- Common in modern APIs

### Plain Java implementation (cursor-based):
```java
public List<Booking> fetchAllBookings() throws InterruptedException {
    List<Booking> all = new ArrayList<>();
    String cursor = null;

    do {
        String url = buildUrl(cursor);
        PartnerResponse response = callWithRetry(url);
        all.addAll(response.getBookings());
        cursor = response.getNextCursor();

        if (cursor != null) {
            Thread.sleep(100); // respect rate limits between pages
        }
    } while (cursor != null);

    return all;
}
```

### Resumability on failure:
```java
// save progress so you can resume from last successful page
public List<Booking> fetchResumable(String savedCursor) {
    String cursor = savedCursor; // null = start from beginning
    do {
        persistCursor(cursor);   // save before each page — resume here if crash
        // fetch page...
    } while (cursor != null);
}
```

### Senior points:
- Always add delay between pages — avoid hitting rate limits
- Persist cursor/page position for large jobs — resume on failure instead of restart
- Page size trade-off: larger pages = fewer requests but higher memory pressure per request
- Some partners have max page size limits — check documentation
