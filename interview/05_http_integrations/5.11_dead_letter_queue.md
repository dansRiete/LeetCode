# 30. How do you implement a dead letter queue in plain Java without a framework?

## Core Idea
A dead letter queue (DLQ) is a holding area for messages/records that failed processing
after all retry attempts. Instead of losing them or crashing, you park them for
manual review or reprocessing later.

---

## Plain Java Implementation

```java
public class DeadLetterQueue<T> {

    private final Queue<FailedRecord<T>> queue = new ConcurrentLinkedQueue<>();

    public record FailedRecord<T>(
        T record,
        String errorMessage,
        int attemptCount,
        Instant failedAt
    ) {}

    public void add(T record, String errorMessage, int attemptCount) {
        queue.offer(
            new FailedRecord<>(record, errorMessage, attemptCount, Instant.now())
        );
        // in production: alert if queue size exceeds threshold
        if (queue.size() > 100) {
            log("WARNING: DLQ size exceeded 100 — investigate partner failures");
        }
    }

    public List<FailedRecord<T>> drainAll() {
        List<FailedRecord<T>> records = new ArrayList<>();
        FailedRecord<T> record;
        while ((record = queue.poll()) != null) {
            records.add(record);
        }
        return records;
    }

    public int size() {
        return queue.size();
    }
}
```

## Usage with Retry Logic
```java
public void processWithRetry(List<String> records, DeadLetterQueue<String> dlq) {
    for (String record : records) {
        int attempts = 0;
        boolean success = false;

        while (attempts < MAX_RETRIES && !success) {
            try {
                process(record);
                success = true;
            } catch (TransientException e) {
                attempts++;
                Thread.sleep((long) Math.pow(2, attempts) * 100); // backoff
            } catch (PermanentException e) {
                dlq.add(record, e.getMessage(), attempts); // don't retry permanent
                break;
            }
        }

        if (!success && attempts == MAX_RETRIES) {
            dlq.add(record, "Max retries exceeded", attempts);
        }
    }
}
```

---

## What to Store in Each DLQ Record
- The original record/message — to allow reprocessing
- Error message and exception type — to understand why it failed
- Attempt count — to know how hard you tried
- Timestamp — to know how long it's been sitting there
- Partner/source identifier — to know which integration failed

---

## In Production (GCP Context)
In production you wouldn't use an in-memory queue — you'd use:
- **GCP Pub/Sub dead letter topic** — built-in DLQ support
- **Cloud Storage** — write failed records to a GCS bucket for manual review
- **BigQuery** — write failed records for analysis

---

## Senior Points to Mention
- In-memory DLQ is lost on restart — only suitable for prototyping
- Always alert when DLQ receives messages — silence is not acceptable
- Have a runbook for how to reprocess DLQ messages
- DLQ size is a key metric to monitor — sudden spike = partner issue
- Distinguish between DLQ (give up after retries) vs retry queue (will retry later)
