## Complete Answer

### Queue vs Topic:
| | Queue | Topic |
|-|-------|-------|
| Model | Point-to-point | Pub/Sub |
| Consumers | One receives each message | All subscribers get every message |
| Use case | Payment processing | Order confirmed → billing + inventory + notifications |
| AWS equiv | SQS | SNS |

### Acknowledgement modes:
| Mode | When | Risk |
|------|------|------|
| AUTO_ACKNOWLEDGE | After receive() returns | Message lost if processing fails |
| CLIENT_ACKNOWLEDGE | Manual — message.acknowledge() | Safest — ack after successful processing |
| DUPS_OK_ACKNOWLEDGE | Lazy/batched | Duplicates possible |

Safe pattern:
```java
try {
    processMessage(msg);
    msg.acknowledge();
} catch (Exception e) {
    session.recover(); // redeliver
}
```

### Persistent vs Non-persistent:
- **Persistent** — written to disk, survives restart. Use for payments, bookings.
- **Non-persistent** — memory only, lost on crash. Use for real-time prices, metrics.

### JMS vs Kafka:
| | JMS | Kafka |
|-|-----|-------|
| Storage | Deleted after consumption | Immutable log, retained for days |
| Replay | No | Yes — consumers track offset |
| Throughput | Moderate | Millions/sec |
| Use case | Task queues, routing | Event streaming, audit, replay |
