## Complete Answer

### Stack vs Heap:

- Stack stores: primitives, object references;
  Heap stores: actual objects
- Stack scope: per thread, per method call;
  Heap scope: shared across all threads
- Stack size: small, fixed;
  Heap size: large, configurable (-Xmx)
- Stack lifetime: until method returns;
  Heap lifetime: until GC collects
- Stack error: StackOverflowError;
  Heap error: OutOfMemoryError

### Heap generations:
```
Heap
├── Young Generation
│   ├── Eden (new objects allocated here)
│   ├── Survivor S0
│   └── Survivor S1
└── Old Generation (Tenured)
    └── Long-lived objects promoted here
```

### GC behavior:
- **Minor GC** — cleans Young Generation, frequent, fast
- **Major/Full GC** — cleans Old Generation, infrequent, slow, causes pauses

### Object promotion:
- New object → Eden
- Survives Minor GC → Survivor space
- Survives several GCs → promoted to Old Generation

### Metaspace (Java 8+):
- NOT part of the heap
- Stores class metadata, method definitions
- Replaced PermGen — no longer fixed size, grows dynamically

### Why it matters for integrations:
- Holding large response payloads too long → promoted to Old Generation
- Too much Old Gen pressure → Full GC pauses → service appears unresponsive to partners
- Connection pools, caches are long-lived → live in Old Generation
