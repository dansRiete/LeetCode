## Complete Answer

### Confirm with GC logs first:
```bash
# Java 9+
-Xlog:gc*:file=gc.log:time,uptime,level

# Java 8
-XX:+PrintGCDetails -XX:+PrintGCDateStamps -Xloggc:gc.log
```
Look for long `Pause Full` entries matching your spike timing.

### Other useful flags:
- `-XX:+HeapDumpOnOutOfMemoryError` — auto heap dump on OOM
- `-Xms` == `-Xmx` — prevent heap resizing which triggers extra GCs

### Two most common root causes of frequent Full GC:
1. **High allocation rate** — tight loops, n+1 queries, creating many short-lived objects
   → floods Young Gen → premature promotion to Old Gen
2. **Old Generation accumulation** — long-lived objects (static caches, session state) fill Old Gen
   → Full GC triggered when Old Gen exhausted

### Fixes:
- Bound caches with eviction (Guava/Caffeine, or LinkedHashMap with max size)
- Review unbounded static collections
- Reduce object creation in hot paths (reuse, pooling)
