## Complete Answer

### GC roots (starting points for reachability tracing):
- Local variables on any active thread's stack
- Static fields of loaded classes
- Active threads themselves
- JNI references (native code holding Java objects)

### Memory leak in Java — yes, possible:
An object is never collected as long as any GC root can reach it.
A memory leak = holding a reference to an object longer than needed.

### Classic examples:
```java
// 1. Unbounded static collection
static List<byte[]> cache = new ArrayList<>();
void handleRequest() {
    cache.add(new byte[1024 * 1024]); // never removed → OOM
}

// 2. Listener never unregistered
eventBus.register(listener); // forgotten → listener + its references stay alive

// 3. ThreadLocal not cleaned up in thread pool
threadLocal.set(heavyObject); // thread returned to pool, value survives → wrong data + leak
```

### Rule of thumb:
- GC cannot save you from **logical** leaks — objects are alive *by definition* if referenced
- Always remove from collections, unregister listeners, and call `threadLocal.remove()`
