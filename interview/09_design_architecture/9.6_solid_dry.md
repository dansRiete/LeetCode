## Complete Answer

### SOLID:
| | Principle | One-liner |
|-|-----------|-----------|
| S | Single Responsibility | One reason to change |
| O | Open/Closed | Open for extension, closed for modification |
| L | Liskov Substitution | Subtypes must honor parent contract |
| I | Interface Segregation | Don't force unused methods on implementors |
| D | Dependency Inversion | Depend on abstractions, not concretions |

### LSP violation — immutable collections:
```java
List<String> list = List.of("a", "b");
list.add("c"); // UnsupportedOperationException — contract broken at runtime
```

### OCP violation — if/else per partner type:
```java
// BAD — every new partner modifies existing class
if (type.equals("STRIPE")) { ... }
else if (type.equals("PAYPAL")) { ... }

// GOOD — new partner = new class, zero changes to existing code
interface PaymentPartner { void process(Payment p); }
class StripePartner implements PaymentPartner { ... }
class PaypalPartner implements PaymentPartner { ... }
```

### ISP violation — fat connector interface:
```java
// BAD — ReadOnlyPartner forced to implement createOrder/cancelOrder
interface PartnerConnector {
    Order getOrder(String id);
    void createOrder(Order o);
    void cancelOrder(String id);
    byte[] downloadInvoicePdf(String id);
}

// GOOD — split by capability
interface OrderReader { Order getOrder(String id); }
interface OrderWriter { void createOrder(Order o); void cancelOrder(String id); }
```

### Note: ISP violations often cause LSP violations
Fat interfaces → implementors throw UnsupportedOperationException → LSP broken
