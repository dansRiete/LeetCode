## Complete Answer

### What it is:
`ReentrantLock` is an explicit lock from `java.util.concurrent.locks` that provides
the same mutual exclusion as `synchronized` but with additional capabilities.
"Reentrant" means the same thread can acquire it multiple times without deadlocking
(same as synchronized).

### Basic usage:
```java
ReentrantLock lock = new ReentrantLock();

lock.lock();
try {
    // critical section
} finally {
    lock.unlock(); // ALWAYS in finally — unlike synchronized, not released automatically
}
```

### Key difference — synchronized releases automatically, ReentrantLock does NOT:
```java
// synchronized — monitor always released even on exception
synchronized(this) {
    throw new RuntimeException(); // monitor released automatically
}

// ReentrantLock — MUST call unlock() manually, always in finally
lock.lock();
try {
    throw new RuntimeException();
} finally {
    lock.unlock(); // without this — lock is never released → deadlock
}
```

---

### Capabilities synchronized does NOT have:

**1. Interruptible lock acquisition:**
```java
// synchronized — thread waits forever, cannot be interrupted
// ReentrantLock — thread can be interrupted while waiting
try {
    lock.lockInterruptibly(); // throws InterruptedException if thread is interrupted
} catch (InterruptedException e) {
    // handle cancellation
}
```

**2. Timeout on lock acquisition:**
```java
if (lock.tryLock(5, TimeUnit.SECONDS)) {
    try {
        // got the lock within 5s
    } finally {
        lock.unlock();
    }
} else {
    // couldn't acquire lock in time — handle gracefully, no deadlock
}
```

**3. Non-blocking try:**
```java
if (lock.tryLock()) { // returns immediately — true if acquired, false if not
    try { ... }
    finally { lock.unlock(); }
} else {
    // lock is held by someone else — do something else
}
```

**4. Fairness — first come, first served:**
```java
// unfair (default) — any waiting thread can grab the lock, faster but may starve
ReentrantLock unfair = new ReentrantLock();

// fair — threads acquire in the order they requested it, prevents starvation
ReentrantLock fair = new ReentrantLock(true);
```

**5. Multiple conditions (like multiple wait/notify queues):**
```java
ReentrantLock lock = new ReentrantLock();
Condition notFull  = lock.newCondition();
Condition notEmpty = lock.newCondition();

// producer waits on notFull, consumer waits on notEmpty
// more precise than synchronized which has only one wait queue per object
```

---

### When to use which:

- Simple critical section → `synchronized` — less code, harder to misuse
- Need to interrupt waiting thread → `ReentrantLock`
- Need timeout on lock acquisition → `ReentrantLock`
- Need try-without-blocking → `ReentrantLock`
- Need fairness guarantee → `ReentrantLock(true)`
- Multiple wait conditions → `ReentrantLock` + `Condition`

### Rule of thumb:
- Default to `synchronized` — simpler, no risk of forgetting `unlock()`
- Reach for `ReentrantLock` only when you need one of the above capabilities

### In integration context:
- `tryLock(timeout)` — useful when acquiring a resource that may be contended,
  fail fast instead of waiting forever
- `lockInterruptibly()` — useful in long-running background threads that need to
  respond to shutdown signals
