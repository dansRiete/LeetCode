# 10. How do you handle backpressure when consuming a high-volume external API?

## Your Answer

## What Was Missing / Wrong

## Complete Answer

### What is backpressure:
When a **producer sends data faster than the consumer can process it**.
Example: partner streams 10,000 events/sec, your connector processes 1,000/sec —
backlog grows, eventually OutOfMemoryError.

### Strategy 1 — Bounded queue + blocking (most common in plain Java):
```java
BlockingQueue<String> queue = new ArrayBlockingQueue<>(1000);

// producer blocks when queue is full
queue.put(record);       // blocks if full — natural backpressure

// consumer processes at its own pace
String record = queue.take(); // blocks if empty
process(record);
```

### Strategy 2 — Rate limiting on consumer side:
```java
ScheduledExecutorService scheduler = Executors.newScheduledThreadPool(1);
// process max 100 records per second
scheduler.scheduleAtFixedRate(() -> {
    String record = queue.poll();
    if (record != null) process(record);
}, 0, 10, TimeUnit.MILLISECONDS);
```

### Strategy 3 — Batching:
```java
// accumulate records, process in batches — reduces per-record overhead
List<String> batch = new ArrayList<>();
queue.drainTo(batch, 100); // drain up to 100 at once
processBatch(batch);
```

### Strategy 4 — Circuit breaker:
- If consumer is overwhelmed, stop accepting new work temporarily
- Signal producer to slow down or pause

### Senior points:
- Monitor **queue depth** as a metric — alert when approaching capacity
- Distinguish between **dropping messages** (acceptable for metrics/logs) vs
  **blocking** (required for financial/booking data)
- For GCP: Cloud Pub/Sub handles backpressure natively — consumers pull at their
  own pace
- `ArrayBlockingQueue` (bounded) preferred over `LinkedBlockingQueue` (unbounded) —
  bounded prevents OOM
