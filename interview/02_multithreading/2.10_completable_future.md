## Complete Answer

### CompletableFuture vs Future:
- Future: blocking only (get()), no chaining, no callbacks, no exception handling
- CompletableFuture: non-blocking callbacks, chainable, composable, manually completable

### thenApply vs thenCompose vs thenCombine:
- `thenApply` — sync transform, like map(): `order -> order.getId()`
- `thenCompose` — async chain, like flatMap(): next step also returns CF
- `thenCombine` — zip two independent futures: `(order, customer) -> ...`

### Fan-out pattern:
```java
List<CompletableFuture<Order>> futures = partnerIds.stream()
    .map(id -> CompletableFuture.supplyAsync(() -> fetchOrder(id), executor))
    .toList();

List<Order> results = CompletableFuture.allOf(futures.toArray(new CompletableFuture[0]))
    .thenApply(v -> futures.stream().map(CompletableFuture::join).toList())
    .get(10, TimeUnit.SECONDS);
```
- allOf() takes array not list
- join() is unchecked, cleaner inside streams
- Always use custom executor for I/O, not ForkJoinPool.commonPool()

### Exception handling:
| Method | Called when | Changes result | Use for |
|--------|------------|----------------|---------|
| exceptionally | Exception only | Yes (fallback) | Default value on failure |
| handle | Always | Yes | Transform both success and error |
| whenComplete | Always | No | Logging, metrics side effects |

```java
// Fallback to cache on partner failure
CompletableFuture.supplyAsync(() -> partnerApi.getPrice(id))
    .exceptionally(ex -> cachedPrice(id))
    .thenAccept(price -> displayPrice(price));
```
