## What Was Missing / Wrong
- **Waiting threads CANNOT be interrupted** — biggest practical limitation, not mentioned
- Missing: no timeout support — thread waits forever with no way out
- These two gaps are exactly why `ReentrantLock` is preferred in production

## Complete Answer

### How it works:
```java
// on a method
public synchronized void process() { ... }

// on a block — more granular, preferred
public void process() {
    synchronized(this) { ... }
}

// on a static method — locks the Class object
public static synchronized void process() { ... }
```

### Reentrancy:
```java
// safe — same thread can re-enter without deadlocking
public synchronized void methodA() {
    methodB(); // calls another synchronized method on same object — fine
}
public synchronized void methodB() { ... }
```

### Monitor always released on exception:
```java
synchronized(lock) {
    throw new RuntimeException(); // monitor still released — JVM guarantees this
}
```

### Limitations:

- Cannot interrupt waiting thread: synchronized has no fix;
  ReentrantLock fix: `lockInterruptibly()`
- No timeout on lock acquisition: synchronized has no fix;
  ReentrantLock fix: `tryLock(5, TimeUnit.SECONDS)`
- Cannot try without blocking: synchronized has no fix;
  ReentrantLock fix: `tryLock()`
- No fairness guarantee: synchronized has no fix;
  ReentrantLock fix: `new ReentrantLock(true)`

### When to prefer ReentrantLock:
```java
ReentrantLock lock = new ReentrantLock();

// interruptible wait
lock.lockInterruptibly(); // throws InterruptedException if thread interrupted

// timeout
if (lock.tryLock(5, TimeUnit.SECONDS)) {
    try { ... }
    finally { lock.unlock(); }
} else {
    // gave up waiting — handle gracefully
}
```
