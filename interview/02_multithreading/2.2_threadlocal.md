## Complete Answer

### What it is:
```java
ThreadLocal<String> correlationId = new ThreadLocal<>();

correlationId.set("req-123");        // set for current thread
correlationId.get();                 // "req-123" — only visible to this thread
correlationId.remove();              // clean up
```

### Concrete integration use cases:
1. **Correlation/trace ID** — attach to every log without passing through every method
2. **Auth context** — current API key or user for the duration of a request
3. **Transaction context** — DB connection tied to current request thread

```java
// set once at request entry point
RequestContext.set(new Context(correlationId, apiKey));

// available anywhere in the call chain without passing as parameter
void processBooking() {
    log.info("Processing - correlationId: {}", RequestContext.get().correlationId());
}
```

### The memory leak danger in thread pools:
```java
// Thread pool reuses threads — ThreadLocal values persist between requests
// Request 1 sets value
threadLocal.set(user1Context);
// Thread returned to pool

// Request 2 picks up same thread
threadLocal.get(); // returns user1Context — WRONG USER, security issue!
```

### The fix — always clean up:
```java
try {
    threadLocal.set(value);
    processRequest();
} finally {
    threadLocal.remove(); // CRITICAL — always in finally block
}
```

### Rule of thumb:
- Use `ThreadLocal` for **cross-cutting concerns** that shouldn't pollute method signatures
- **Always** `remove()` in a `finally` block when using thread pools
- Prefer `InheritableThreadLocal` if child threads need to inherit the value
