## Complete Answer

### What volatile guarantees:
- **Visibility** — any write is immediately visible to all threads (no stale CPU cache)
- Does NOT guarantee **atomicity** of compound operations

### When volatile is sufficient:
```java
// single writer, multiple readers — volatile is enough
volatile boolean running = true;

// Thread 1 — worker
while (running) {
    doWork();
}

// Thread 2 — controller
running = false; // immediately visible to Thread 1
```

### When volatile is NOT sufficient:
```java
volatile int counter = 0;
counter++; // NOT atomic — read + increment + write = 3 separate operations
// two threads can read same value simultaneously → lost update

// fix — use AtomicInteger
AtomicInteger counter = new AtomicInteger(0);
counter.incrementAndGet(); // atomic
```

### Decision table:

- One writer, many readers → `volatile`
- Simple boolean flag → `volatile`
- Multiple writers → `synchronized` or `Atomic*`
- Compound check-then-act → `synchronized`
- Counter increments → `AtomicInteger`

### volatile vs synchronized:

- Visibility: both provide it
- Atomicity: volatile does not; synchronized does
- Blocking: volatile never blocks; synchronized does
- Performance: volatile is faster; synchronized is slower
