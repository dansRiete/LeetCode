# 1.5 LinkedList vs ArrayDeque — which to use as a stack?

## Key Point
`ArrayDeque` is almost always the better choice over `LinkedList` for stack/queue use.

## Why LinkedList is worse:
- Allocates a **node object per element** — extra memory, GC pressure
- Nodes scattered in heap — poor **cache locality**, slower traversal
- `peek()` reads from front, `peekLast()` reads from back — easy to mix up
  and cause subtle bugs (as seen in MinStack)

## Why ArrayDeque is better:
- Backed by a **resizable array** — no per-element allocation
- Better **cache locality** — elements stored contiguously in memory
- `push`/`pop`/`peek` all consistently operate on the **same end** — no ambiguity

```java
// prefer this
Deque<Integer> stack = new ArrayDeque<>();
stack.push(1);       // adds to front
stack.pop();         // removes from front
stack.peek();        // reads front — always consistent

// over this
Deque<Integer> stack = new LinkedList<>();
stack.add(1);        // adds to back
stack.removeLast();  // removes from back
stack.peekLast();    // easy to accidentally call peek() instead — wrong end
```

## When LinkedList IS appropriate:
- Frequent **insertions/deletions in the middle** of the list
- You need both `Queue` and `Deque` semantics on the same structure
- You need `null` elements (ArrayDeque does not allow nulls)

## Rule of thumb:
- Stack or queue → `ArrayDeque`
- Need index-based access → `ArrayList`
- Need frequent middle insertions → `LinkedList`
