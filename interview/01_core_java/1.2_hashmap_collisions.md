# 2. How does HashMap handle collisions internally?

## Your Answer
- Collision happens when hashCode returns same value for different objects ✓
- Uses equals() as second check ✓
- Stores colliding entries in a list ✓
- Degrades performance ✓

## What Was Missing / Wrong
- Collision happens when two keys map to the **same bucket**
  (same `hashCode % bucketCount`), not necessarily same hashCode value
- It's not just a "list" — it's a **linked list**, and since Java 8 converts to a
  **red-black tree** at 8+ entries
- Missing: **load factor** (0.75 default) and **rehashing**
- Missing: must override both `equals()` AND `hashCode()` on keys — if equal objects
  have different hashCodes they land in different buckets and `get()` never finds
  what you `put()`
- Tree converts back to linked list at **6 entries** (gap of 2 prevents thrashing)

## Complete Answer

### Bucket lifecycle:
1. Empty
2. **Linked list** — 1-7 entries, O(n) lookup
3. **Red-black tree** — 8+ entries, O(log n) lookup
4. Back to linked list — drops to 6 entries

### Load factor and rehashing:
- Default capacity: **16 buckets**
- Default load factor: **0.75**
- Rehash threshold: 16 × 0.75 = **12 entries**
- When reached: doubles to 32 buckets, redistributes all entries

### equals/hashCode contract:
```java
// BROKEN — equal objects in different buckets, get() never finds them
class BadKey {
    int id;
    public boolean equals(Object o) { return this.id == ((BadKey)o).id; }
    // missing hashCode override — uses Object.hashCode() (memory address)
}

// CORRECT
class GoodKey {
    int id;
    public boolean equals(Object o) { return this.id == ((BadKey)o).id; }
    public int hashCode() { return Integer.hashCode(id); }
}
```

### Values — equals/hashCode never called:
- Only **keys** use equals/hashCode
- Values are just stored and retrieved
- Exception: `containsValue()` — linear scan, rarely used
