# 24. How do you handle gRPC deadlines/timeouts vs HTTP timeouts?

## What Was Missing / Wrong
- Key concept: gRPC uses **deadlines** (absolute time), not timeouts (relative duration)
- Most important difference: **deadline propagation** across the call chain

## Complete Answer

### HTTP timeouts — socket level, no propagation:
```java
conn.setConnectTimeout(3_000);  // 3s to connect
conn.setReadTimeout(10_000);    // 10s to read response
// server has no idea about your timeout — keeps processing after you give up
```

### gRPC deadlines — propagated across entire call chain:
```java
// client sets deadline
stub.withDeadlineAfter(5, TimeUnit.SECONDS)
    .getFlightStatus(request);
```

### The key difference — propagation:
```
HTTP:
Client (10s timeout) → Service A → Service B
If client times out, Service A and B keep running — wasted work

gRPC:
Client sets 5s deadline → Service A (knows: 3s left) → Service B (knows: 1s left)
If deadline passes anywhere in chain → entire chain cancels
```

### Server must check for cancellation:
```java
public void streamFlights(Request request, StreamObserver<FlightResponse> observer) {
    for (Flight flight : flights) {
        if (Context.current().isCancelled()) {  // check before each item
            observer.onError(Status.CANCELLED.asException());
            return; // stop processing — no wasted work
        }
        observer.onNext(buildResponse(flight));
    }
    observer.onCompleted();
}
```

### Rules:
- Always set deadlines — never leave unset (default = wait forever)
- Set on the **client side** — server respects it
- Make deadline realistic — too short = false failures, too long = defeats purpose
- Deadline is **absolute time**, not duration — accounts for time already spent
  in the chain

### Summary:

- HTTP Timeout type: relative duration;
  gRPC Deadline type: absolute point in time
- HTTP propagation: no;
  gRPC propagation: yes — across entire chain
- HTTP server awareness: no;
  gRPC server awareness: yes
- HTTP wasted server work: yes;
  gRPC wasted server work: no (cancelled)
- HTTP configuration: per connection/read;
  gRPC configuration: per call
