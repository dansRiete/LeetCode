## Complete Answer

### Core differences:

- Protocol: REST uses HTTP/1.1; gRPC uses HTTP/2
- Data format: REST uses JSON (text); gRPC uses Protobuf (binary)
- Schema: REST schema is optional (OpenAPI); gRPC requires a .proto file
- Browser support: REST yes; gRPC no (needs gRPC-Web proxy)
- Human readable: REST yes; gRPC no
- Performance: REST slower; gRPC 3-10x faster, smaller payload
- Streaming: REST limited; gRPC built-in (4 patterns)
- Code generation: REST optional; gRPC built-in from .proto

### HTTP/2 advantages gRPC gets for free:
- **Multiplexing** — multiple requests over one connection
- **Header compression** — reduces overhead
- **Bidirectional streaming** — both sides can stream simultaneously

### Browser limitation:
Browsers cannot directly call gRPC because HTTP/2 at browser level doesn't expose
the low-level controls gRPC needs. Solution: **gRPC-Web** proxy layer.

### When to choose:

**Choose REST when:**
- Public-facing API (browser clients)
- Human readability matters (easy debugging)
- Partner doesn't support gRPC
- Simple request/response, no streaming

**Choose gRPC when:**
- Internal microservice-to-microservice communication
- High performance / low latency required
- Streaming data (live updates, large datasets)
- Strong contract enforcement needed
- Polyglot environment (auto-generated clients in any language)

### Memory trick:
- REST = **human friendly** (readable JSON, browser compatible)
- gRPC = **machine friendly** (binary, fast, strongly typed)
