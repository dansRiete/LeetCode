# 27. How do you version a Protobuf schema without breaking existing clients?

## Your Answer
- Based on field numbers not names ✓

## What Was Missing / Wrong
- Correct foundation
- Missing: `reserved` keyword — most important safety mechanism
- Missing: what changes are dangerous (type changes, number reuse)
- Missing: package-level versioning for breaking changes

## Complete Answer

### Foundation — field numbers, not names:
```protobuf
message Flight {
    int32 passenger_id = 1;   // clients use number 1, not "passenger_id"
    string destination = 2;
}
```
Renaming `passenger_id` to `id` — safe. Wire format unchanged.

### Safe changes:
```protobuf
message Flight {
    int32 passenger_id = 1;
    string destination = 2;
    string currency = 4;      // added new field — old clients ignore unknown numbers ✓
}
```

### Dangerous changes:
```protobuf
// NEVER reuse a field number
message Flight {
    int32 passenger_id = 1;
    // destination was field 2 — if you reuse 2 for something else:
    int32 price = 2;  // WRONG — old clients will deserialize price as destination!
}

// NEVER change field type
string passenger_id = 1;  // was int32, now string — corrupts old clients
```

### The reserved keyword — prevents accidental reuse:
```protobuf
message Flight {
    int32 passenger_id = 1;
    // destination removed — reserve BOTH number and name
    reserved 2;
    reserved "destination";
    string currency = 3;
}
// now field 2 and name "destination" can never be reused — compile error if attempted
```

### Breaking changes — version at package level:
```protobuf
// v1 — keep running for existing clients
package integration.flights.v1;
message Flight { ... }

// v2 — breaking change (removed required field, changed type, etc.)
package integration.flights.v2;
message Flight { ... }
```
Run both versions simultaneously → deprecate v1 once all clients migrated.

### Safe vs unsafe summary:

- Add new field (new number) → safe
- Rename a field → safe
- Remove a field + reserve number → safe
- Add new RPC method → safe
- Remove a field without reserving → NOT safe
- Reuse a field number → NOT safe
- Change field type → NOT safe
- Change field number → NOT safe
