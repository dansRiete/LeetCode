# 26. What happens when a gRPC server streams data and the client disconnects?

## Your Answer
- Not sure

## What Was Missing / Wrong
- Key point: server must actively check `isCancelled()` — it won't stop automatically

## Complete Answer

### What happens automatically:
- gRPC detects disconnection via HTTP/2 connection state
- Server receives a **cancellation signal**
- `Context.current().isCancelled()` returns true
- If server is blocking on `StreamObserver.onNext()` — throws
  `StatusRuntimeException(CANCELLED)`

### What does NOT happen automatically:
- A server loop does NOT stop on its own — must check cancellation explicitly
- Without check: server keeps processing and sending data to nobody —
  **wasted resources**

### Server must check isCancelled():
```java
public void streamFlights(Request request, StreamObserver<FlightResponse> observer) {
    for (Flight flight : getAllFlights()) {

        // check before each item — stop if client disconnected
        if (Context.current().isCancelled()) {
            observer.onError(Status.CANCELLED
                .withDescription("Client disconnected")
                .asException());
            cleanupResources(); // close DB connections, cancel partner calls
            return;
        }

        observer.onNext(FlightResponse.newBuilder()
            .setFlight(flight)
            .build());
    }
    observer.onCompleted();
}
```

### Why this matters in production:
```
Without isCancelled() check:
- 1000 clients connect, each triggers a stream of 10000 records
- 500 clients disconnect mid-stream
- Server still processes 500 × remaining records
- Thread pool exhausted, memory pressure, partner API overloaded
```

### Resource cleanup on cancellation:
```java
if (Context.current().isCancelled()) {
    partnerApiCall.cancel();      // cancel outbound partner calls
    dbConnection.close();         // release DB connections
    observer.onError(...);
    return;
}
```

### Adding a cancellation listener:
```java
Context.current().withCancellation().addListener(
    context -> log("Client cancelled stream — cleaning up"),
    executor
);
```

### Summary:
- HTTP/2 detects disconnection automatically ✓
- Server must **actively check** `isCancelled()` ✓
- Always clean up resources on cancellation ✓
- gRPC deadlines make this more powerful — cancellation propagates to downstream
  calls too
