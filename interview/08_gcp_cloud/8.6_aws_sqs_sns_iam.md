# 8.6 — AWS: SQS vs SNS, FIFO vs Standard, IAM

## Complete Answer

### SNS vs SQS:
| | SNS | SQS |
|-|-----|-----|
| Model | Push, fan-out | Pull, queue |
| Consumers | Many simultaneously | One per message |
| Persistence | No | Yes (up to 14 days) |
| Use case | Notify multiple systems | Async job processing with retry |

Classic pattern: SNS → multiple SQS queues (fan-out to independent consumers)

### SQS Standard vs FIFO:
| | Standard | FIFO |
|-|----------|------|
| Delivery | At-least-once (duplicates possible) | Exactly-once |
| Ordering | Best-effort | Guaranteed |
| Throughput | Unlimited | 300 msg/s |
| Use case | Idempotent, order-irrelevant | Payment steps, ordered sequences |

### IAM:
- **Policy** — JSON document defining allowed/denied actions on resources
- **User** — human or app with static long-lived credentials (avoid in app code)
- **Role** — assumed by services (EC2, Lambda, ECS), generates temporary credentials

Golden rule: never hardcode access keys — always use IAM roles with temporary credentials.

```
ECS Task Role → SQS:ReceiveMessage, S3:GetObject, SecretsManager:GetSecretValue
```
